SECURITY AUDIT REPORT
Standalone Vite Vanilla JS SPA
Repository: C:\Users\vdako\toomuchwork
Scope: src/main.js, src/vault-api.js, src/style.css, index.html, vite.config.ts, package.json
Deployment target guidance: Cloudflare Pages + Workers

EXECUTIVE SUMMARY
This codebase is a static Vite SPA entirely composed of one 1190-line client-side module (main.js)
with no backend Workers implementation present. It spawns DOM from template strings using
app.innerHTML = shell() plus dynamic innerHTML replacement inside form onsubmit handlers, reads
URL path via location.pathname, secretes a bearer token into window.__VAULT_API_TOKEN__ on the
global object, and simulates payment/sensitive flows without any real validation. The vault-api.js
client is a thin POST wrapper with hardcoded model name "hermes-2-free". There are no other
authentication primitives, rate limits, CSP, or service workers, and there is no manifest/sri logic.

=== FINDINGS ===

-- 1. Input / Injection / XSS

1.1 Global innerHTML shell rendering
Files / lines: src/main.js:910-957
Risk: High
Every navigation re-renders the entire app: app.innerHTML = shell(); and shell() is a 900+ line
template string that interpolates route templates from location.pathname. The path segment is
derived client-side, so a malicious URL fragment or a link that lands on /<img src=x onerror=...>
will be reflected into markup without sanitization. Although Vite history handling mitigates this
specific vector to some degree, the construction is unconditionally unsafe.

1.2 Form onsubmit innerHTML replacement (XSS sink)
Files / lines:
  - src/main.js:158 (checkout)
  - src/main.js:496 (list an item)
  - src/main.js:522 (appraisal)
  - src/main.js:546 (contact/prove/sell form)
Risk: High/Critical
Each form uses inline onsubmit which does:
event.preventDefault();
this.closest('form').innerHTML='<p style=color:#C9A84C;font-family:Cinzel,serif>...';
The form fields are never validated server-side, so an attacker can POST crafted input when the
try-catch is chased to a backend; even currently, these patterns set a precedent where user input
is reflected into innerHTML. Additionally, the inline onsubmit handlers are themselves a bad
practice that can be broken by future sanitization policies.

1.3 URL path used as route key with no allowlist
Files / lines: src/main.js:91
  const path = location.pathname.replace(/^\/+/, '') || 'home';
  ... (routes[path] || routes.home)()
Risk: Medium
Any path value will be looked up in routes and otherwise falls back to home. This does not
execute untrusted content directly but exposes that fragments or locale prefixes could land on an
intended route, and future additions that interpolate path into the DOM can introduce XSS.

1.4 Unsafe global function attachment and browser onclick/onchange handlers
Files / lines: src/main.js:258-259, 467, 779, 897, 1160-1177, etc.
Risk: High
The codebase polllutes window with many functions (_toast, _login, _add, _remove, _toggleW,
_agentRun, _agentCommand, _samsonToggle, _checkout, ...) and calls them from inline HTML
attributes: <button onclick="window._agentCommand('audit','buyer_finder')" ...>. If any of these
functions ever accept raw user input and later inject it into the DOM or an API payload, the lack
of strict validation will enable reflected XSS or command injection against the backend.

1.5 Potential DOM clobbering via library endpoints
Files / lines: src/main.js:760-761
  ${a.endpoint}
  <button onclick="window._agentRun('${a.endpoint}')" ...>
Risk: High
Endpoints are read from a static list, which is fine now, but the path is interpolated from data
and concatenated into a string passed inside quotes without escaping. If endpoints become dynamic
or parser-facing later, an endpoint string containing a single quote or HTML could break out of
the attribute and inject script.

1.6 Shipping quote inputs reflected without sanitization
Files / lines: src/main.js:658-660 (onchange) and 1123-1142 (_shipQuote reads .value)
Risk: Low-Medium
Values flow through to computed text nodes via textContent (safe) now. No DOM insertion of raw
values, but the pattern of reading arbitrary select values and rendering into a cart confirmation
section can become dangerous if future changes interpolate them into innerHTML.

-- 3. Unsafe DOM APIs / innerHTML Usage

3.1 app.innerHTML = shell() (line 957)
Risk: High
Full-page rewrite via innerHTML without any sanitization or escaping. The shell template
interpolates svg strings that are constant and safe, but it also interpolates the current route
path and title strings. Any route path controlled by the user (via URL manipulation) flows into
the DOM without escaping.

3.2 Multiple form innerHTML replacements
Files / lines: src/main.js:158, 496, 522, 546
Risk: High
Each form replaces its own innerHTML with a static success message. While the message is static,
using innerHTML as the mechanism sets precedent for later dynamic messages and can lead to policy
violations with fresh inline script restrictions in Workers/Pages.

3.3 Static SVG strings and social links
Files / lines: src/main.js:8-38 and 40, 118-120
Risk: Low
Stable constants not user-controlled. No immediate XSS risk but the size of svg-in-template
strings adds noise to CSP reviews. Prefer converting SVGs to inline markup or <use> references.

-- 4. CSRF / Auth Token Handling Gaps

4.1 Bearer token from global variable with no CSRF mitigation
Files / lines: src/vault-api.js:13-15, 22-24
Risk: High
The fetch call uses Authorization: Bearer with no CSRF token. Cross-site request forgery is
unprotected because the client does not validate the Origin nor include any CSRF cookie or
double-submit token. Since the token is just a client-side global property, any attacker-controlled
page can make fetch('/api/auth/login', {method:'POST', headers:{Authorization:'Bearer ...'}}).

4.2 No SameSite/HttpOnly cookie usage
Throughout entire codebase
Risk: High
Sessions are not represented by cookies, only by a token passed in JS headers. Cloudflare Pages
and Workers support both same-site cookie enforcement and HttpOnly sessions; none are used.

4.3 No request signing, nonce, or idempotency
src/vault-api.js:19-31
Risk: Medium
All mutations are plain POSTs with no idempotency key; replay or double-submit via browser back/
refresh is not protected at the client layer.

4.4 Token persistence in IndexedDB without expiry checking
Files / lines: src/main.js:1082-1083, 1069-1070, 1091-1092
Risk: Medium
state.user is persisted in IndexedDB, including the result.session token presumably. There is no
expiry check, no refresh, and no validation on restore(); an expired token would continue to be
sent.

-- 5. Data Validation Gaps

5.1 No client-side schema validation on any forms
Files / lines: src/main.js:159-165 (checkout), 497-514 (sell), 547-555 (appraisal/contact)
Risk: High
User-entered fields (address, card number text, description / provenance) are accepted as plain
strings with no regex/Luhn/email/URL checks. Because the same-on-submit handler does not
touch the values, they are not transmitted to an API in this stub, but any future call using
these inputs will need server-side validation before storing or processing.

5.2 Shipping quote parsing assumes monetary format
Files / lines: src/main.js:1132-1133
  const itemText = itemEl?.textContent || 'Selected item';
  const m = String(itemText).match(/(\$\s*[0-9.,]+)/);
  const itemRaw = m ? Number(m[1].replace(/[^0-9.]/g, '')) : 0;
Risk: Low
Regex is naive; it precedes fragile behavior if UI strings change. No actual financial input is
accepted here, so it is not a direct attacker vector, but is brittle.

=== REMEDIATIONS FOR Cloudflare Pages + Workers ===

R1) Replace global window.__VAULT_API_TOKEN__ with Cloudflare Pages-managed env
    that is inlined at build time (via import.meta.env) into a fetch wrapper that sets
    headers directly, not via global var. Pair with a Cloudflare Worker proxy that sets
    an HttpOnly SameSite=Strict session cookie; on the client, remove Authorization header
    and rely on cookies + CSRF token.

R2) Eliminate inline onsubmit/onclick and innerHTML.
    - Replace app.innerHTML = shell() with either a DOM assembly step using document.createElement
      and textContent or a lightweight template library such as lit-html / lit with
      directive-level escaping.
    - For forms, attach event listeners via addEventListener; for success messages, replace via
      textContent or an existing toast element rather than innerHTML.
    - For the shell/nav/interpolated ids, use textContent and DOMPurify if any rich text
      rendering is required.

R3) Allowlist URL paths in a real router map early.
    Replace (routes[path] || routes.home)() with a whitelist-based match. Render a 404 for
    unknown paths and never reflect raw path segments into the DOM.

R4) Add Content-Security-Policy.
    In Cloudflare Pages, configure _headers (static asset) or the Worker to emit a strict CSP:
      default-src 'self'; script-src 'self' https://accounts.google.com https://appleid.cdn-apple.com;
      img-src 'self' https://images.unsplash.com data:; connect-src 'self' https://api.workers.cloudflare.com;
      frame-src 'none'; object-src 'none'; base-uri 'self'.
    Add nonce support if inline scripts remain for the particle canvas or toasts; treat inline
    onclick strings as removed.

R5) Move checkout / sensitive forms into Worker-validated POST endpoints.
    - Invalidate card inputs at the Blade/Worker layer using a STRIPE_PUBLISHABLE + intent flow;
      never store full card numbers.
    - Enforce schema validation with zod/valibot in the Worker before any downstream logic.

R6) Service worker for caching.
    A Pages-compatible service worker (via workbox-precaching) can cache the shell. Do not rely
    on stale API responses that carry auth tokens.

R7) Remove public email; route contact through a ticket form on Workers.
    ratchetkrewelabs@gmail.com appears in multiple places; replace with a Workers-backed
    /contact route that uses a reCAPTCHA-like verification (Cloudflare Turnstine) and forwards
    to support email.

R8) External image integrity.
    For Unsplash CDN images, add crossorigin="anonymous" + referrerpolicy="no-referrer" and
    consider hosting images on a Cloudflare R2 bucket with strict cache-control and signed URLs.

R9) Sanitize npm input in during _agentCommand and _agentRun.
    Define an explicit action allowlist and route allowlist; never concatenate user-supplied
    values into query strings or text without encoding.

R10) Audit and lock vite.config.ts.
    - Remove server.host and preview.host overrides (or override to localhost in dev, only
      dist in Pages). Set build.outDir = 'dist'; ensure publicDir removed or harmless.
    - Provide a separate production environment config that does not enable source maps unless
      required for debugging.

R11) Enforce HTTPS and HSTS.
    Pages provides HTTPS by default; ensure custom domains force TLS and HSTS via the Worker.

-- PRIORITIZED ACTION ITEMS

P0) Remove Authorizaton header from client and migrate to HttpOnly cookie sessions via
    Cloudflare Worker. Remove all window.__VAULT_API_TOKEN__ references and never place
    tokens in LocalStorage, IndexedDB, or window globals.

P1) Eliminate every innerHTML assignment in src/main.js. Replace with safe DOM APIs or
    lit-html-style tagged templates with auto-escaping.

P2) Move all inline onsubmit/onclick attributes to addEventListener handlers and route IDs
    through a strict allowlist router derived in src/main.js:91.

P3) Add CSP and SRI headers in the Pages deployment config/_headers.

P4) Validate all financial inputs, addresses, and endpoints with strict parsing in the Worker
    and client.

-- CONCLUSION

The current code is a static shell/DOM renderer with a stub API client and authentication state
persisted to IndexedDB + window globals. For a production Cloudflare Pages + Workers rollout,
token handling must be moved to HttpOnly cookies and the SPA renderer must abandon innerHTML in
favor of safe DOM construction. The rest of the remediation aligns with standard Cloudflare
patterns (Turnstile, HSTS, signed image URLs, zod-validated Workers POSTs).
