EXECUTIVE SUMMARY
The current codebase is a modern full-stack application running on Cloudflare Pages with Hono,
tRPC, Drizzle ORM (D1), Clerk/OAuth + custom local auth, Stripe payments, and React 19.
Significant security improvements exist over the legacy SPA (strong CSP, HSTS, rate limiting,
parameterized queries, proper JWT handling). However, the audit identified Critical findings
in authorization, CSRF, information disclosure, and session management that require immediate
attention before production use.

=== FINDINGS ===

-- 1. CRITICAL SECURITY RISKS

1.1 Unauthenticated order modification (Broken Access Control)
Files: api/orders-router.ts:51-81
Risk: Critical
The orders.updateStatus mutation is publicQuery — ANY unauthenticated user can update the
paymentStatus, orderStatus, or trackingNumber of ANY order by ID. There is no ownership
check, no authentication, and no authorization. An attacker can:
  - Mark orders as "paid" / "delivered" to falsify records
  - Inject tracking numbers to fraud shipping confirmations
  - Cancel/refund orders arbitrarily
Recommendation: Change to authedQuery and add ownership check:
  if (order.userId !== ctx.user?.id && ctx.user?.role !== "admin") throw FORBIDDEN

1.2 Unauthenticated order creation (Broken Access Control / Abuse)
Files: api/orders-router.ts:20-49
Risk: Critical
orders.create is publicQuery. Any unauthenticated user can create unlimited orders with
arbitrary amounts, titles, and shipping addresses. This enables:
  - Database pollution / DoS via massive order creation
  - Fraudulent order records that can later be "confirmed" via 1.1
Recommendation: Require authentication. Either authedQuery or validate against active
Stripe session before allowing creation.

1.3 Debug endpoint exposing environment variable names (Information Disclosure)
Files: functions/[[path]].ts:193-198
Risk: Critical
GET /api/env-keys returns all environment variable names sorted, including:
  - SOL_USD_RATE, USDC_USD_RATE, SOLANA_TREASURY, TREASURY_WALLET
  - OPENAI_API_KEY, OPENAI_BASE_URL, CLERK_SECRET_KEY, CLERK_WEBHOOK_SIGNING_SECRET
  - STRIPE_SECRET_KEY, COINBASE_API_KEY, GOOGLE_CLIENT_SECRET, GITHUB_CLIENT_SECRET
While values are masked by the env system, leaking the full surface area reveals:
  - Which services are integrated (Solana, Stripe, OpenAI, Clerk, Coinbase)
  - Exact secret variable names for targeted attacks
  - Treasury wallet configuration
Recommendation: Remove /api/env-keys entirely from production. If diagnostics are needed,
protect behind adminQuery + audit log, and redact sensitive key names.

1.4 Local auth token transmitted via custom header (CSRF vulnerability)
Files: api/local-auth-router.ts, api/context.ts, src/providers/trpc.tsx
Risk: Critical / High
The local auth system stores JWT tokens in sessionStorage/localStorage and sends them via
the X-Local-Auth-Token header. Browsers automatically attach custom headers to same-origin
requests. If any XSS exists (see 2.1), an attacker can make the victim's browser send
authenticated requests to ALL tRPC endpoints including mutations. Because CORS has
credentials: true, cross-origin requests with custom headers will include cookies, but the
local auth token itself is a bearer token in a header with no CSRF protection.
Additional issue: The fallback JWT secret ("fallback-secret") in env.ts:48 means if
APP_SECRET is unset, any attacker can forge valid local auth tokens.
Recommendation:
  - Migrate local auth to HttpOnly SameSite cookies (like OAuth already does)
  - Remove x-local-auth-token header approach
  - Remove the APP_SECRET fallback — fail closed if the secret is missing

1.5 In-memory audit logs and token revocation (Compliance / Security gap)
Files: api/security.ts:230-294
Risk: High
Audit logs (auditLog array) and revoked tokens (revokedTokens Set, revokedUserSessions Map)
are stored in process memory. Cloudflare Workers spin down after inactivity. On cold start:
  - All audit history is lost (compliance violation)
  - Revoked tokens are no longer recognized (compromised tokens remain valid)
  - User session revocations are wiped (can't force logout after password change)
Recommendation: Persist audit logs and revocation state to D1 database. At minimum, store
revoked token jti claims and user revocation timestamps in D1.

-- 2. FUNCTIONAL BUGS

2.1 Legacy dead code still in repository (src/main.js, src/vault-api.js)
Files: src/main.js, src/vault-api.js
Risk: Medium
The old vanilla JS SPA ( ~1190 lines ) is dead code — index.html only loads main.tsx.
However, these files remain in the repo and could be accidentally deployed or served if
routing changes. They contain innerHTML-based XSS patterns, global token storage, and
unsafe DOM manipulation from the previous architecture.
Recommendation: Delete src/main.js and src/vault-api.js to eliminate confusion and attack
surface.

2.2 process.env.CLOUDFLARE_API_TOKEN in Cloudflare Worker runtime
Files: api/admin-router.ts:214, 236, 259
Risk: High (functional)
Cloudflare Pages Functions (Workers) do NOT have process.env. The admin Cloudflare
management endpoints (cloudflareStatus, cloudflareDeploy, cloudflareBindings) will always
see undefined for apiToken and return errors. This is broken functionality, not a runtime
security hole, but it means Cloudflare management is non-functional from the admin panel.
Recommendation: Use c.env.CLOUDFLARE_API_TOKEN passed through wrangler.toml [vars] or
secrets.

2.3 Stripe webhook secret compared client-side via tRPC public mutation
Files: api/stripe-router.ts:114-160
Risk: High
stripe.handleWebhook is a publicQuery mutation that accepts the Stripe secret key from the
client as an input parameter and compares it to env.stripeSecretKey. While this doesn't
directly leak the secret, it:
  - Allows brute-force / timing attacks on the secret via a public endpoint
  - Normalizes the pattern of sending webhook secrets through the client
  - Bypasses the proper webhook endpoint (functions/[[path]].ts already has a correct
    Stripe webhook handler at /api/stripe/webhook with signature verification)
Recommendation: Remove the tRPC stripe.handleWebhook mutation entirely. The correct
handler is already in functions/[[path]].ts. If client-side webhook simulation is needed,
use a separate admin-only endpoint.

2.4 CORS credentials:true with broad public mutations
Files: api/security.ts:224, api/cart-router.ts, api/orders-router.ts, api/listing-fee-router.ts
Risk: High
The CORS config sets credentials: true, meaning cookies/auth headers are sent cross-origin.
Multiple publicQuery mutations (cart.add, cart.remove, cart.clear, orders.create,
stripe.handleWebhook, email.sendAppraisalResult, newsletter.subscribe) accept requests
without authentication. Combined with credentials:true, an attacker-hosted page can make
the victim's browser send authenticated requests to these endpoints if any XSS exists or
if the user has an active session.
Recommendation: For truly public endpoints that don't need credentials, either:
  - Set credentials: false (but then cookies won't work)
  - Or better: keep credentials:true but require authentication on all mutations; make
    purely public endpoints read-only queries.

2.5 JWT secret fallback to hardcoded string
Files: api/env.ts:32, 48; api/local-auth-router.ts:19-24
Risk: High
If APP_SECRET is not configured, env.appSecret returns "fallback-secret". The
local-auth-router warns but continues. In production, this means ALL JWT tokens are signed
with a known static key, allowing trivial session forgery for any user ID.
Recommendation: Remove the fallback. If APP_SECRET is missing, throw an error during
startup or return 500 for all auth-dependent requests.

2.6 Email content uses user input without HTML sanitization
Files: api/email-router.ts:55-66, api/email-service.ts
Risk: Medium
buildAppraisalEmail interpolates appraisal.itemName, marketAnalysis, etc. directly into
HTML email templates. If itemName or marketAnalysis contains HTML/script (e.g. <script>),
it will be executed in the recipient's email client if the client renders HTML.
Recommendation: Sanitize all user-generated content before inserting into email HTML.
Use a proper HTML sanitizer (DOMPurify via Workers-compatible build, or strip tags server-
side with a strict allowlist).

2.7 Crypto payment status auto-update on getById
Files: api/blockchain-router.ts:154-165
Risk: Medium
blockchain.getById queries a cert by ID and returns it. While seemingly read-only, this
pattern is coupled with other returns. More importantly, there's no authorization — anyone
can query any certificate by ID. This is likely acceptable for public verification but
should be explicitly documented as a public lookup. The real risk is that getByListing
also exposes internal certificate metadata.

2.8 Missing input length validation on several mutation fields
Files: api/orders-router.ts:25-31, api/listings-router.ts
Risk: Low-Medium
listings.create accepts description max 5000 (good) and features max 200 (good). But
orders.create accepts shippingAddress with no maxLength, and listingTitle with no explicit
max in orders. This could lead to oversized payloads / DoS.
Recommendation: Add z.string().max() to all text inputs in mutations.

2.9 Hardcoded Cloudflare account ID in admin router
Files: api/admin-router.ts:214, 237, 260
Risk: Low
Account ID "2ad733f9d698170c202b12924868c60e" is hardcoded. While not a secret, it
couples the admin panel to a single Cloudflare account. Moving it to env is cleaner.

-- 3. OPTIMIZATION / HARDENING OPPORTUNITIES

3.1 Remove 'unsafe-inline' from CSP (if possible)
Files: api/security.ts:159-172
Risk: Optimization
The CSP includes 'unsafe-inline' for scripts. Modern React (via Vite) does not require
inline scripts. Removing 'unsafe-inline' and using nonces/hashes would significantly
reduce XSS impact. This requires auditing all inline event handlers and style attributes.
Note: Hono's own inline script for CORS preflight may need to be excluded via 'script-src
'none' for OPTIONS'.

3.2 Use DOMPurify or equivalent for client-side HTML
Files: src/components/, src/pages/
Risk: Hardening
The sanitizeInput function in api/security.ts uses regex-based XSS pattern replacement.
This is bypassable (e.g., encoded tags, mutation XSS). For any user content rendered in
React, prefer React's auto-escaping (JSX text nodes). For any rich-text rendering, use
DOMPurify. The server-side sanitizeInput should be replaced or supplemented.

3.3 Enable source maps only for non-production builds
Files: vite.config.ts:27
Risk: Hardening
sourcemap: false is already set (good). Verify this isn't overridden in production build
scripts or Wrangler config.

3.4 Dependency vulnerabilities
Files: package.json, package-lock.json
Risk: Medium
npm audit reports 9 vulnerabilities:
  - @solana/web3.js <=1.98.4 (moderate, via jayson/uuid)
  - @solana/spl-token, @solana/spl-token-group, @solana/spl-token-metadata (high/moderate)
  - uuid <11.1.1 (moderate)
  - esbuild on Windows dev server (high, dev-only)
The Solana stack is the primary concern. Upgrading @solana/web3.js to the latest patched
version and updating @solana/spl-token to 0.1.8+ will resolve the high/moderate findings.
Action: Run `npm audit fix` and consider manual @solana/* upgrades if semver-major
breaking changes are acceptable.

3.5 Strengthen rate limiting configuration
Files: api/security.ts, functions/[[path]].ts:151-179
Risk: Hardening
Current rate limiting is in-memory Map with max 10,000 entries. In a multi-Cloudflare
Worker environment (if scaled), rate limits are per-instance, not global. For production,
consider:
  - Using Cloudflare KV or D1 for distributed rate limiting
  - Adding per-user rate limits (not just per-IP) for authenticated mutations
  - Applying rate limits to ALL mutations, not just /api/* paths

3.6 Add Content-Security-Policy nonce support
Files: api/security.ts:146-177
Risk: Hardening
The CSP is static. If any inline scripts are unavoidable (Workers runtime polyfills, third-
party widgets), use nonces instead of 'unsafe-inline'. Generate a nonce per request and
inject it into the HTML shell.

3.7 Secure clerk-webhook endpoint routing
Files: src/api/clerk-webhook.ts
Risk: Medium
The Clerk webhook handler is a standalone Hono app but there's no evidence it's mounted
in the main functions/[[path]].ts. If it's not reachable, Clerk events are silently dropped.
Verify it's registered as a Cloudflare Pages Function (e.g., in api/ directory or mounted
in the main app).

3.8 Token revocation not enforced on local auth
Files: api/local-auth-router.ts, api/security.ts:274-294
Risk: Medium
The security module exports isTokenRevoked() and isUserSessionRevoked(), but these are
NEVER called in local-auth-router.ts or context.ts during token verification. Revoking a
token or a user's sessions has no effect because the verification functions don't check
the revocation lists.
Recommendation: Call isTokenRevoked(token) and isUserSessionRevoked(userId, iat) in
verifyLocalTokenAndRefresh and verifyOAuthSessionAndRefresh.

3.9 Hash password with Argon2id or bcrypt if available
Files: api/local-auth-router.ts:271-277
Risk: Optimization
PBKDF2 with 100,000 iterations is acceptable but slower than Argon2id on Workers.
However, Workers don't have native Argon2. Consider using bcryptjs (already in deps) or
upgrade to 200,000+ iterations. Note: crypto.subtle on Workers is fast, but 100k may be
insufficient for modern standards. NIST recommends 600,000 for PBKDF2-SHA256 as of 2023.
Recommendation: Increase to 310,000 iterations (OWASP 2023 baseline for PBKDF2) or switch
to bcrypt with cost factor 12+.

3.10 SQL injection posture
Files: All api/*-router.ts files
Status: Good (no findings)
All queries use Drizzle ORM with parameterized bindings (eq(), sql`` tagged templates, or
.bind()). No raw string concatenation in SQL was observed. Schema is well-defined.

surface. The P1 items address systemic authz and logging gaps. Dependency upgrades are
